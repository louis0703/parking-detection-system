## 问题2：车位检测全部误判为"占用"

**现象**：3张空车位图片全部被判断为"占用"

**原因**：初始代码分析整张图像的方差和边缘密度，但图像中包含墙壁、天花板等干扰元素，导致方差过高

**解决过程**：
1. 第一次尝试：取图像中心50%区域作为ROI → 仍然误判
2. 第二次尝试：取图像下半部分（50%以下）作为ROI → 部分改善
3. 最终方案：增加颜色熵和亮像素占比两个特征，调整阈值组合
   - 方差 > 2500 → 判为占用
   - 边缘密度 > 0.20 → 判为占用
   - 颜色熵 > 5.5 → 判为占用
   - 亮像素占比 > 0.15 → 判为占用

## 问题3：车牌定位检测到15个候选区域（严重误检）

**现象**：第一张车牌图片（奥迪）检测到15个候选区域，大部分是误检

**原因**：HSV颜色范围太宽（H=100~130, S>50, V>50），把车身上的蓝色装饰、天空反射等都检测出来了

**解决**：
1. 收紧HSV范围：H=100~125, S>80, V>80
2. 增加面积过滤：候选区域不能超过图像面积的10%
3. 增加尺寸过滤：高度>30像素，宽度>80像素

## 问题4：第二张车牌图片检测到错误区域（挡风玻璃）

**现象**：丰田图片（京K·BT355）的蓝色掩膜同时检测到了车牌和挡风玻璃区域，最终选中了挡风玻璃

**原因**：
- 挡风玻璃反射天空呈蓝色，被HSV检测捕获
- 挡风玻璃面积远大于车牌，按面积排序时被优先选中

**解决**：修改评分算法，综合考虑三个因素：
1. **长宽比评分**：越接近3.14（标准车牌比例）分数越高
2. **位置评分**：y坐标越大（越靠下）分数越高，因为车牌通常在图像下半部分
3. **面积惩罚**：面积过大（>图像2%）时扣分，避免选中挡风玻璃

**评分公式**：
```
score = ratio_score × position_score × area_penalty × area
```

## 问题8：车位7（有施工材料）误判为"空闲"

**现象**：7.jpg 中有金属围栏、木板、蓝色箱子、橙色锥桶等施工材料占用停车位，但被判断为"空闲"

**原因**：之前的算法基于颜色差异检测非地面物体，但户外网格地面的草缝也产生大量颜色差异，导致：
1. 颜色差异法把整个ROI都标记为"前景"（草缝+物体混在一起）
2. 连通域分析无法区分草缝（小而分散）和物体（大而连续），因为形态学处理后草缝也连成了大片

**尝试过的方法（均失败）**：
1. 颜色距离法（中位数地面色 + 阈值）→ 草缝与物体无法分离
2. 饱和度检测法 → 草缝也是高饱和度（绿色），与物体混淆
3. 频域分析（FFT）→ 网格pattern主导频谱，物体影响太小
4. 纹理分析（局部方差、边缘密度）→ 草缝和物体的纹理特征重叠

**最终方案：棕色物体检测 + 形状分析**
1. 检测棕色像素（H=10-25, S>50, V>50）→ 捕获木板、施工材料
2. 中心区域掩膜（排除边缘墙壁/围栏）
3. 连通域分析 + 形状特征：
   - **关键发现**：草缝中的植被是细长条状（长宽比 > 2.5），而施工材料是紧凑块状（长宽比 < 2.5）
   - 筛选条件：面积 > 中心区域8% **且** 长宽比 < 2.5
4. 判断：有大型紧凑棕色物体 → 占用

## 问题9：车牌字符识别严重不准

**现象**：3张车牌图片识别结果全部有误：
- 1.jpg（京K·BT355）→ 识别为 `就KBt355`（"京"→"就"，多了"t"）
- 2.jpg（苏B92912）→ 识别为 `5B929i2`（漏掉"苏"，"1"→"i"，多了"5"）
- 10.jpg（粤A·0HA88）→ 识别为 `粤a0hA88`（大小写错误）

**原因分析**：

### Bug 1：增强图像未传给 OCR（已修复）
`main.py` 中做了 CLAHE、频域锐化、Unsharp Masking 等增强处理，但 `recognize_plate_with_confidence(plate_roi)` 传入的仍是原始裁剪图 `plate_roi`，增强结果完全没用上。

**修复**：将增强后的图像作为第二个参数传入 OCR 函数。

### Bug 2：ddddocr 模型本身的局限性
- 中文省份字符识别不稳定（"京"→"就"）
- 小尺寸车牌（如 2.jpg 的 150×51 像素）会直接跳过中文字符
- 大小写不分（"a" 不会自动转 "A"）
- 字符 "1" 和 "i" 混淆

### Bug 3：`__pycache__` 缓存导致代码更新不生效
修改了 `_score_plate_text()` 评分函数后，Python 继续加载旧的 `.pyc` 缓存文件，导致省份字符加分逻辑未生效。所有候选结果评分相同（145分），无法选出最优解。

**修复**：清除 `__pycache__` 目录，后续使用 `python -B` 运行。

## 问题10：PaddleOCR 3.x 在当前系统无法运行

**现象**：安装 PaddleOCR 3.6.0 + PaddlePaddle 3.3.1 后，运行报错：
```

[pir::ArrayAttribute<pir::DoubleAttribute>]
```

**原因**：PaddlePaddle 3.x 的 oneDNN 后端与当前 Windows 系统不兼容。

**解决**：改用 `rapidocr-onnxruntime`（v1.2.3），它是基于 PaddleOCR 预训练模型的 ONNX 推理版本：
- 不依赖 PaddlePaddle，无兼容性问题
- 推理速度快（ONNX Runtime）
- 识别精度与 PaddleOCR 一致

## 问题11：RapidOCR 在裁剪图上识别效果差

**现象**：将裁剪后的车牌区域直接传给 RapidOCR，结果只识别出部分字符：
- 1.jpg → `3T355`（漏掉前3个字符）
- 10.jpg → `OHA88`（漏掉前3个字符）
- 2.jpg → `929`（漏掉大部分字符）

**原因**：RapidOCR 的流程是 **文字检测 → 文字识别**。在裁剪的小图上，文字检测阶段无法正确找到文字区域，导致只识别了部分字符。

**解决**：改用 **全图识别 + 区域匹配** 策略：
1. 对完整原始图像运行 RapidOCR（检测+识别）
2. 根据车牌区域坐标过滤结果，只保留中心点落在车牌区域内的文本
3. 容差设为区域尺寸的 20%，防止因定位偏差导致漏匹配

**效果**：3张图片全部正确识别。

## 问题12：停车场10.jpg（白色车）误判为"空闲"

**现象**：10.jpg 中一辆白色五菱宏光停在车位上，但被判断为"空闲"

**原因**：之前的检测算法只通过 HSV 颜色检测"棕色物体"（木板、施工材料等），白色/银色/黑色车辆不在检测范围内。

**数据验证**：

通过对比分析有车和无车场景的视觉特征，发现关键区别：

| 特征 | 10.jpg（有车） | 4.jpg（空网格） | 9.jpg（夜间空） |
|------|---------------|----------------|----------------|
| 边缘密度 | 0.070 | 0.258 | 0.025 |
| 局部方差 | 88 | 1380 | 35 |
| 低方差块比例 | 45.9% | 11.5% | 84.0% |

**核心发现**：空网格地面的每个块都有高对比度（亮砖+暗洞），而车漆面的块是低对比度的均匀区域。

**最终方案：纹理遮挡检测（补充棕色检测）**

在原有棕色物体检测基础上，新增基于块对比度的纹理分析：

1. 将中心区域分成 32×32 的块
2. 计算每个块的对比度（max - min）
3. 对比度 < 40 的块标记为"低对比度块"
4. 闭运算连接相邻的低对比度块
5. 连通域分析找最大连通域
6. 判断规则：
   - 最大连通域占中心区域 5%~40% → 可能是车
   - 中心区域整体亮度 > 60（排除夜间暗区和室内暗地面）
   - 棕色检测 OR 纹理检测 任一命中 → 判为占用

**各场景区分效果**：

| 场景 | 最大连通域占比 | 亮度 | 检测结果 |
|------|:---:|:---:|------|
| 10.jpg 白色车 | 10.8% | 89 | 纹理命中 → **占用** ✅ |
| 7.jpg 施工材料 | 0%（棕色检测命中） | 90 | 棕色命中 → **占用** ✅ |
| 4/5/6/8.jpg 空网格 | 0-3% | 79-109 | 均未命中 → **空闲** ✅ |
| 9.jpg 夜间空 | 54.9%（>40%排除） | 39（<60排除） | 排除 → **空闲** ✅ |
| 1/2.jpg 室内空 | 79-86%（>40%排除） | 74-90 | 排除 → **空闲** ✅ |
